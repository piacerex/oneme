# oneme ROADMAP

Ready Player Me のような「Webで作成でき、複数アプリから利用できる3Dアバター生成サービス」を目指す。
最初から完全互換を狙わず、まずはパーツ選択式の軽量MVPを作り、早い段階で顔写真からの初期アバター提案とAI生成MVPを追加する。
SDK、VRM対応、商用運用は後続フェーズで拡張する。

## 目標

- ユーザーがWeb上で3Dアバターを作成できる
- 作成したアバターをアカウントに保存できる
- 外部アプリがAPI経由でアバター情報やモデルURLを取得できる
- `.glb` を基本形式として配信し、必要に応じて `.vrm` に対応する
- Unity / Web / Three.js から簡単に読み込める

## 想定ユースケース

- メタバース、VR/AR、Web3Dサービス向けの共通アバター
- UnityゲームやWebアプリに組み込めるユーザーアバター
- 社内イベント、バーチャル展示会、ライブ配信向けの簡易アバター
- 顔写真や画像生成AIと連携した似顔アバター生成

## 方針

- Phase 1では手動作成を成立させ、Phase 2で「顔写真から初期アバター案を作る」体験を前倒しする
- Phase 2の顔写真対応は完全自動生成ではなく、肌色、髪色、顔パーツの推薦から始める
- 旧Phase 7の高度なAI生成はPhase 3へ前倒しし、最初は複数スタイル案とテクスチャ候補のMVPに絞る
- 最初は高品質な少数アセットで、確実に動く作成体験を優先する
- サーバー側で巨大な3D処理を抱えすぎず、まずは設定JSON + パーツURLで成立させる
- 合成済みモデル出力は後続フェーズで追加する
- API、埋め込みWidget、SDKの順に外部連携を広げる

## 推奨技術スタック

- Frontend: React / Next.js / Three.js / React Three Fiber
- Backend: Phoenix / Node.js / Hono など
- Database: PostgreSQL
- Storage: S3互換ストレージ
- CDN: Cloudflare / CloudFront
- 3D format: `.glb` first, `.vrm` later
- 3D tooling: Blender Python, gltf-transform, meshoptimizer
- Auth: Email login, OAuth
- API: REST first, GraphQL optional

### 実装ステップ

## Phase 2: 顔写真から初期アバター提案

### 機能

- 顔写真アップロード
- 顔写真の一時保存と自動削除
- 顔写真をプレビューへ一時的に参照表示
- 肌色の推定
- 髪色の推定
- 顔の輪郭や雰囲気に近い顔パーツの推薦
- 推薦結果から初期アバターを生成
- ユーザーが手動で補正できるUI
- 顔写真を使わずに作成する導線

### 方針

- 完全な3D顔復元は行わない
- 写真から直接メッシュを生成せず、既存パーツの中から近いものを選ぶ
- 顔写真のプレビュー反映は参照表示に留め、アバターの顔メッシュへ直接貼り込まない
- 似顔精度よりも「すぐ始められる」「あとで直せる」体験を優先する
- 顔写真は初期提案にだけ使い、保存し続けない

### データモデル

- face_analysis_jobs
- face_analysis_results
- avatar_recommendations

### API

- `POST /api/face_analysis_jobs`
- `GET /api/face_analysis_jobs/:id`
- `POST /api/avatars/from_face_analysis`

### 注意点

- 顔写真の保存ポリシーを明確にする
- 削除機能を必ず用意する
- 未成年やセンシティブ情報の扱いに注意する
- 顔写真を使わない作成方法を常に用意する

### 完了条件

- 顔写真から初期アバター案を生成できる
- 顔写真を一時的な参照としてプレビュー上で確認できる
- ユーザーが生成結果を編集して保存できる
- 顔写真が一定時間後に自動削除される

### 実装ステップ

1. 顔写真アップロードUIを作る
2. 写真利用の同意文と削除方針を表示する
3. アップロード画像を一時保存する
4. 顔写真をプレビューへ一時的に参照表示する
5. 顔解析ジョブを作成する
6. 肌色、髪色、顔パーツ候補を推定する
7. 推定結果を既存パーツIDにマッピングする
8. 推薦結果からアバター設定JSONを生成する
9. ユーザーが推薦結果を編集できる画面へ遷移する
10. 顔写真の自動削除ジョブを実装する
11. 顔写真なしの通常作成フローと同じ保存APIへ接続する

## Phase 3: AI生成MVP

### 機能

- Phase 2の顔解析結果をAI生成入力として拡張
- 髪型、顔パーツ、メイク、服装の高度な自動提案
- 画像生成AIによるテクスチャ候補生成
- 複数スタイルのアバター案を同時生成
- 生成結果の安全チェック
- ユーザーが生成案を選んで手動編集できるUI

### 方針

- 最初は3Dメッシュ生成ではなく、既存パーツへの推薦とテクスチャ候補生成に絞る
- 顔写真そのものを長期保存せず、解析結果と生成履歴だけを必要最小限で扱う
- 本人性が強すぎる生成やセンシティブ属性推定を避ける
- 生成案は必ずユーザー編集画面を通して保存する

### データモデル

- ai_generation_jobs
- ai_generation_candidates
- ai_texture_candidates
- recommendation_feedback

### API

- `POST /api/ai_generation_jobs`
- `GET /api/ai_generation_jobs/:id`
- `POST /api/avatars/from_ai_candidate`
- `POST /api/recommendation_feedback`

### 注意点

- 顔写真の保存ポリシーを明確にする
- 削除機能を必ず用意する
- 画像生成AIで扱う入力情報と禁止情報を定義する
- 生成結果の審査と拒否フローを作る
- 生成結果が本人に似すぎる場合の利用規約を整える

### 完了条件

- 複数のアバター案を生成できる
- 画像生成AIによるテクスチャ候補を安全に適用できる
- ユーザーが生成結果を編集して保存できる

### 実装ステップ

1. Phase 2の顔解析結果をAI生成用の入力スキーマへ拡張する
2. 画像生成AIで扱う入力情報と禁止情報を定義する
3. テクスチャ生成のプロンプトテンプレートを作る
4. 複数スタイルの候補生成UIを作る
5. 生成結果の審査と拒否フローを作る
6. 生成済みテクスチャを既存パーツへ適用する
7. ユーザー編集履歴を推薦改善に使うデータ構造を作る
8. 生成結果の再現性とキャッシュ方針を決める
9. プライバシー、削除、監査ログを強化する
10. 本人性や過度な類似に関する利用規約を更新する

## Phase 4: GLBエクスポート

### 機能

- 選択済みパーツを1つの `.glb` として出力
- エクスポートジョブ管理
- 生成済みモデルのキャッシュ
- モデル軽量化
- テクスチャ最適化

### 3D処理

- Blender Pythonでパーツ合成
- gltf-transformで最適化
- Dracoまたはmeshoptimizerによる圧縮
- LODまたは低ポリ版の生成

### 完了条件

- APIから合成済み `.glb` URLを取得できる
- Unity / Three.js で読み込める
- 生成処理が非同期ジョブとして安定して動く

### 実装ステップ

1. エクスポートジョブのDBスキーマを作る
2. アバター設定JSONから必要パーツを解決する
3. Blender Pythonの合成スクリプトを作る
4. 合成前後のモデル検証コマンドを用意する
5. gltf-transformによる最適化処理を追加する
6. 生成済み `.glb` をストレージへ保存する
7. エクスポート状態とモデルURLを返すAPIを作る
8. 同じ設定JSONの場合は既存モデルを再利用する
9. 生成失敗時のエラー情報と再試行を実装する
10. UnityとThree.jsの最小ビューアで読み込み検証する

## Phase 5: 埋め込みWidget

### 機能

- 外部サービスに埋め込めるiframe版アバター作成Widget
- 作成完了時のcallback
- アプリごとのテーマ設定
- アプリごとの許可パーツ制御
- API key管理

### API

- `POST /api/apps`
- `POST /api/apps/:id/api_keys`
- `GET /widget?app_id=...`
- `postMessage` による作成完了通知

### 完了条件

- 外部HTMLに数行でアバター作成画面を埋め込める
- 作成完了後にavatar idを呼び出し元へ返せる

### 実装ステップ

1. アプリ登録とAPI key管理のDBスキーマを作る
2. Widget用の最小レイアウトを作る
3. `app_id` とAPI keyで利用元アプリを判定する
4. アプリごとの許可パーツ設定を適用する
5. 作成完了時に `postMessage` でavatar idを返す
6. iframe埋め込み用のサンプルHTMLを作る
7. CORSと埋め込み許可ドメインを制御する
8. Widgetテーマ設定を反映する
9. 作成途中の離脱と再開を扱う
10. 外部ページから作成、保存、再取得までを通しで検証する

## リスク

- 3Dアセット制作コストが高い
- パーツ同士の干渉や破綻が起きやすい
- リギングとアニメーション互換が難しい
- 顔写真やAI生成はプライバシー対応が必要
- `.glb` と `.vrm` の両対応は検証コストが高い
- 商用アセットのライセンス管理を怠ると後で使えなくなる

## MVPで割り切ること

- 完全な3D顔復元と独自メッシュ生成は後回し
- Phase 2では顔写真から既存パーツを推薦するだけにする
- Phase 3ではAI生成をテクスチャ候補と既存パーツ推薦に絞る
- パーツ数は少なくてよい
- 最初は `.glb` のみ
- SDKは後回しで、まずはREST APIとiframe
- サーバー合成前に、設定JSON方式で動かす
